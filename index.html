    Setting up git version control with github

    Git is a pretty simple, fast, yet powerful version control tool. It means you can keep track of all changes you make to a project and revert to earlier versions at any time. This gives you the peace of mind that everything you do is reversable, and is indispensable when working on live websites. You can also branch your code to do a side project in parallel and merge them together later, and allows for multiple people to work on the same project without worrying about interfereing with each other.

    Github is an online community of git users that allows easy online code sharing and project collaboration.

    How to set up git version control on your project.

    • Install git, I'm using Ubuntu, so I will type this into the terminal sudo apt-get install git
    • Tell git your username and email - this information is saved with every commit and is especially important when sharing code. The following sets the global configuration, which will be used for everything you do. It is possible to set this for individual projects as well. git config --global user.name "Your Name" git config --global user.email "your_email@example.com"
    • Go to the working directory you would like to version control cd ~/workspace/myproject
    • Initialise an empty reposotory (repo) git init
    • Now you have to add the files you want to control in the staging area ready to be committed, this must be done before every commit. To add everything in the directory: git add .
    • You can check the status of files that have changed, and add or remove files before committing. git status
    • Finally, you commit them to the repo, with a description of the changes you are committing for later reference. git commit -m "description" You need to do a commit every time you want to save the current state of the project, if you don't commit regularly, it becomes useless.
    • Note: if files have been deleted and you want to remove them from the repositary, use the -u option, then commit. git add -u .

    Setting up a github account.

    • Go to github.com and sign up for an account
    • In order to connect to your account and upload your repo you will need a SSH key. Here is how to create a new one: ssh-keygen -t rsa -C "your_email@example.com" ssh-add id_rsa Choose a passphrase
    • Now you can give your public key to github. Copy the contents of ~/.ssh/id_rsa.pub xclip -sel clip < ~/.ssh/id_rsa.pub Go to your github account > account settings > SSH keys > add SSH key, and then paste in the key and save it.
    • To see if its working you can try to SSH into github ssh -T git@github.com Enter your passphrase, then you should see the following message: Hi [username]! You've successfully authenticated, but GitHub does not provide shell access.
    • Next you need to create a repo on github to upload your project to. Go you your github account, select 'create a new repo', and fill in the required information. Your repo will have a HTTPS clone URL, copy this.
    • Now go back to your terminal in your project directory, and you will tell it where the online repositary is, and call it 'origin' git remote add origin [HTTPS clone URL]
    • Fnally you will push our git repo up to github git push origin master
    • If there are changes in the github repo, (for example you created a readme file,) you will need to pull the changes to your local repo before you can push your files up git pull origin

    Copying a Joomla! website using Akeeba backup and restore

    These are the simple steps I should have taken today, if I had done it the easy way.

    The task, to put a Joomla! website online from a .jpa file created with the Joomla extention Akeeba backup, working from the command line with SSH and SCP to interact with a server running Debian.

    • Create a fresh install of Joomla! with a MySQL database
      • Create a new MySQL database and user with a password, and grant all permissions to that user
      • Extract (unzip) the Joomla! download file into the web root on the server (for example into '/var/www/mysite/')
      • Go to the new Joomla! homepage by typing in the domain name (or server ip if you don't have a domain yet) of your website into your browser (eg www.yourdomain.com), and follow the instructions that should appear to install it.
    • Use Akeeba Kickstart to unpack the .jpa file and copy the website over the Joomla installation
      • Extract the Kickstart download somewhere, and copy kickstart.php into your web root (the same place you extracted the Joomla file to)
      • Place the .jpa file in the same directory
      • Open the kickstart file in your browser by typing www.yourdomain.com/kickstart.php and follow the instructions to restore it. When connecting to the database select the option to drop any tables already in the database, or do it manually later
      • Make sure the '$live_site' in the Joomla configuration.php file is pointing to the right place (eg www.mydomain.com)

    The basics of search engine optimisation (SEO)

    SEO is a huge topic, and companies invest huge amounts of money and effort researching and developing ways to improve their rankings on search engines. This makes it seem a very intimidating and intractable topic to hopeful young web developers like myself. But actually the basics are pretty simple, and mostly common sense.

    The single most important thing is to have good content, content that users will want to find, will come back to, and will recomend to their friends. Without good content, no amount of SEO is going to improve your rankings.

    Design for users, not for search engines. There's been a kind of arms race between search engines and websites trying to increase traffic in any way they can. For example in the beginning search engines used helpful keyword information embedded in the html <meta> tags to determine the content of websites, but malicious websites would use that to promote their website by stuffing it with popular keywords that were nothing to do with the content of the website, and so the search engines had to get smarter. They are now very good at matching what uses are looking for to the actual content of a site, while also penalising sites that try to manipulate them.

    However, search engines read your website as plain text, they can't see the amazing graphics, JavaScript special effects, or CSS that you might have created to lead the user around the page and highlight important information. (there are also some users who will prefer to view websites this way too). If you only had the HTML to your page, would your site still be understandable by a user? Have a look at your website through tools like seo-browser.com to try this out.

    • Make sure all images have useful alt attributes to display when the images can't be seen.
    • Your HTML should have a good structure: make sure you are making correct use of heading tags <h1> to <h6>, and in HTML5 the new structural tags such as <heading>, <section>, and <article>.
    • Tip: Images can go in <h1> elements, and the alt text is seen as the heading when the image isn't displayed, this way you can have your fancy heading graphic, while also maintaing the document structure to be picked up by search engines.
    • You need to make sure your HTML, CSS and JavaScript code is valid and following best practices to help the search engine do its job. You could try W3C’s validation tools, Lint, and JSHint to check your site.

    Lets talk about keywords. Keywords are just as important for a user browsing your webpage looking for information, as they are for search engine rankings. Again, if you design for the user, the search engines will take care of themselves.

    What appears in search engine results is the title and meta description of your page, so make sure these are sensible to ensure people will click you when you do appear in search engines.
